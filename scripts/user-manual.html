Introduction & Overview
What MarketSync is and what it can do

1.1 What is MarketSync?

MarketSync is a full-stack, real-time multi-market trading dashboard built on Next.js 16. It lets registered users monitor live price data from three major global exchanges — India (NSE), United States (NYSE / NASDAQ), and Hong Kong (HKEX) — and execute paper trades (simulated orders) with a virtual starting balance of ₹1,00,000.

All price data is streamed over WebSocket connections. Candlestick (OHLCV) data is persisted in a dedicated TimescaleDB time-series database for fast historical queries, while transactional user data (accounts, orders, positions, alerts) is stored in a Prisma Postgres managed PostgreSQL database.

✅
Default Market: India (NSE)

After login, users are redirected to the India watchlist. The India market and ₹ (Rupee) currency are shown by default throughout the application.

1.2 Key Features

📊 Live Market Data

  • Real-time NSE price feeds via Upstox WebSocket
  • US & HK prices via AllTick WebSocket
  • Bid / Ask / Spread / Mid price display
  • Market status clock (timezone-aware)
  • Connection health badges

📈 Candlestick Charts

  • Interactive OHLCV charts (lightweight-charts)
  • Multiple timeframes: 1m, 5m, 15m, 30m, 1h, 1d
  • Period filters: 1W, 1M, 3M, 6M, 1Y
  • TimescaleDB caching (stale-while-fresh)
  • Auto-fallback to daily interval when market closed

🛒 Paper Trading

  • Market & Limit order types
  • Buy / Sell with balance validation
  • Weighted-average cost tracking
  • Order history with status tracking
  • Multi-market order management

💼 Portfolio & P&L

  • Live unrealized P&L per position
  • Realized P&L ledger on each sale
  • Total portfolio value (cash + holdings)

  • Filter by market (All / India / US / HK)

🔔 Price Alerts

  • Conditions: ABOVE / BELOW / EQUAL

  • Full alert history & status tracking

👤 User Profile

  • Initials avatar with account stats
  • Edit display name
  • Change password (bcrypt-secured)
  • Account metadata (ID, email, join date)

⚙️

  • pg Pool (TimescaleDB)

🗄️

  • → Users, Orders, Positions, Alerts, PnL
  • TimescaleDB (TigerData cloud)
  • → Candle OHLCV time-series

2.2 Database Architecture (Dual-DB)

MarketSync uses a split database strategy for optimal performance:

DatabaseProviderStoresAccess Method
Prisma Postgres Prisma Cloud (db.prisma.io) User, Order, Position, PnlEntry, PriceAlert Prisma ORM (@prisma/client)
TimescaleDB TigerData Cloud Candle (market, symbol, interval, time, open, high, low, close, volume) Raw SQL via pg Pool (lib/timescale.ts)
ℹ️
Why Two Databases?

Prisma Postgres does not support the TimescaleDB extension. TimescaleDB is purpose-built for time-series data with automatic hypertable partitioning (30-day chunks), columnar compression, and fast range queries — ideal for OHLCV candle storage and retrieval.

3.3 Database Setup

Prisma Postgres (Relational Data)

  1. Create a project at prisma.io and copy the connection strings.
  2. Add DATABASE_URL and DIRECT_DATABASE_URL to .env.
  3. Run migrations: npx prisma migrate deploy
  4. Generate the Prisma client: npx prisma generate

TimescaleDB (Candle Data)

  1. Create a TimescaleDB instance at tigerdata.com (or Timescale Cloud).
  2. Copy the PostgreSQL connection string to TIMESCALE_URL in .env.
  3. Initialize the Candle hypertable by running the provided SQL script:
    psql "$TIMESCALE_URL" -f scripts/init-timescale.sql
⚠️
Candle Caching is Optional

If TIMESCALE_URL is empty, the app still works — it will fetch candles from Upstox / AllTick on every request without caching. Set TIMESCALE_URL for production to avoid rate limits and improve chart load times.

Authentication
Sign up, login, JWT sessions, and route protection

4.1 Sign Up

New users register at /signup. The registration form requires:

  • Full Name — displayed in the profile avatar and navigation
  • Email Address — must be unique; used for login
  • Password — hashed with bcryptjs (10 rounds) before storage

On successful registration, the user receives a ₹1,00,000 starting balance and is redirected to the India watchlist.

ℹ️
Password Security

Passwords are never stored in plain text. bcrypt hashing with a cost factor of 10 is applied before the record is written to the database. Password comparison on login uses bcrypt.compare().

4.2 Login

Existing users log in at /login with their email and password. On success:

  1. Server verifies the password hash with bcrypt.
  2. A JWT token is signed using JWT_SECRET with a 7-day expiry.
  3. The token is set as an httpOnly cookie named ms_token.
  4. The browser is redirected to /india.

To log out, visit any page and click the logout option in the navigation. This calls POST /api/auth/logout which clears the ms_token cookie.

import { getSession } from '@/lib/session';

export async function GET(request: NextRequest) {

Paper Trading Engine
Placing orders, execution logic, and position tracking

6.1 Placing Orders

Navigate to /orders to place trades. Orders can also be initiated by clicking Buy or Sell on any watchlist row, which pre-fills the order form.

Order Form Fields

  • Marketindia / us / hk
  • SymbolInstrument code (e.g., RELIANCE, AAPL)
  • SideBUY or SELL
  • Order TypeMARKET (execute immediately) or LIMIT (wait for price)
  • QuantityNumber of units to buy or sell
  • PriceLimit price (for LIMIT orders) or current market price (for MARKET)
  • NotesOptional memo / strategy note

6.2 Order Types & Statuses

Order Types

  • MARKET — Executes instantly at the current price. Balance is deducted / credited immediately.
  • LIMIT — Saved as PENDING. Not executed until manually filled or a matching price trigger occurs.

Order Statuses

  • PENDING — Awaiting execution (LIMIT)
  • FILLED — Fully executed
  • PARTIAL — Partially filled
  • CANCELLED — Cancelled by user
  • REJECTED — Insufficient funds or position

6.3 Order Execution Logic

The paper trading engine lives in lib/trading.ts. Here is how each trade is processed:

BUY Order

  1. Calculate total cost: quantity × price.
  2. Check user balance is sufficient. If not → REJECTED.
  3. Deduct cost from user balance.
  4. Create or update a Position record using weighted-average cost:
    newAvgCost = (oldQty × oldAvgCost + newQty × price) / (oldQty + newQty)
  5. Mark order as FILLED with avgFillPrice and filledAt timestamp.

SELL Order

  1. Check that a Position exists for this symbol with sufficient quantity. If not → REJECTED.
  2. Credit proceeds (quantity × price) to user balance.
  3. Calculate realized P&L: (price - avgCost) × quantity.
  4. Create a PnlEntry record for the realized gain/loss.
  5. Reduce position quantity. If position reaches zero → delete position record.
  6. Mark order as FILLED.
ℹ️
No Interactive Transactions

Prisma Postgres does not support interactive transactions via the pg adapter. All write operations are performed as sequential queries rather than wrapped in a single database transaction.

7.2 P&L Calculation

Unrealized P&L

Calculated live using current market price:

pnl = (price - avgCost) × qty
pnl% = (pnl / costBasis) × 100

Updates in real-time as prices change in the WebSocket feed. Shown in green (profit) or red (loss).

Realized P&L

Recorded when a SELL order executes:

realizedPnl =
  (salePrice - costBasis)
  × quantity

Each sale creates a permanent PnlEntry record. The Realized P&L table shows full sell history.

Price Alerts
Set price triggers for automatic notifications or trade execution

Navigate to /alerts to manage price alerts. Alerts monitor price levels and fire notifications or auto-execute trades when conditions are met.

Creating an Alert

FieldOptionsDescription
SymbolAny tracked instrumentThe stock to monitor
Marketindia / us / hkWhich exchange
ConditionABOVE · BELOW · EQUALPrice direction trigger
Target PriceNumericPrice level that triggers the alert
ActionNOTIFY · BUY · SELLWhat happens when alert fires

Alert Conditions

Trigger Logic

  • ABOVE — fires when currentPrice > targetPrice
  • BELOW — fires when currentPrice < targetPrice
  • EQUAL — fires when price is within ±0.5% of target

Alert Actions

  • NOTIFY — browser notification (no trade)
  • BUY — auto-executes a MARKET buy order
  • SELL — auto-executes a MARKET sell order

Alert Status Lifecycle

StatusMeaning
ACTIVEMonitoring — will fire when price condition is met
TRIGGEREDCondition was met; action was taken. Stored for history.
CANCELLEDManually cancelled by user before triggering

Alert Trigger Endpoint

The endpoint POST /api/alerts/check accepts a price payload and evaluates all ACTIVE alerts for matching symbols. It is designed to be called by an external price-checking service or scheduled job.

// POST /api/alerts/check
{
  "symbol": "RELIANCE",
  "market": "india",
  "price": 2500.00
}
⚠️
Auto-Trade Alerts

When an alert with action BUY or SELL triggers, it executes a real paper trade on your account. Ensure your balance and positions are adequate before setting BUY/SELL alerts.

Data Providers & Market Data
How live prices and historical candles are sourced

Provider Overview

MarketProviderProtocolData Type
India (NSE)UpstoxWebSocket (protobuf) + REST v2Live ticks + historical OHLCV
US (NYSE/NASDAQ)AllTickWebSocket (JSON) + RESTLive order book + historical OHLCV
Hong Kong (HKEX)AllTickWebSocket (JSON) + RESTLive order book + historical OHLCV

Upstox (India)

Live Price Feed

  • OAuth 2.0 authorization generates a bearer token stored as UPSTOX_ACCESS_TOKEN.
  • The token is used to request an authorized WebSocket URL from /api/upstox/ws-url.
  • Price updates arrive as protobuf-encoded binary messages, decoded by lib/upstox-tick-data/.
  • The decoded tick includes: LTP (last traded price), bid, ask, volume, OI.

Historical OHLCV

Upstox provides two REST endpoints used by /api/india/kline:

Endpoint TypeURL PatternUsed For
Historical /v2/historical-candle/{instrument}/{unit}/{toDate}/{fromDate} Multi-day / multi-week historical data (1d, 1w, 1mo intervals)
Intraday /v2/historical-candle/intraday/{instrument}/{unit} Today's intraday candles only (1m–1h when no period specified)
ℹ️
Upstox Data Direction

Upstox returns candles in newest-first order. The kline route reverses the array to ascending chronological order before returning or caching.

AllTick (US / HK)

Live Price Feed

  • AllTick provides a single WebSocket endpoint for US and HK markets.
  • Messages are JSON with order book depth (bid/ask levels).
  • A heartbeat mechanism keeps the connection alive. Auto-reconnect on disconnect.
  • Up to 300 price history points are maintained per symbol for sparklines.

Historical OHLCV (REST)

GET https://quote.alltick.co/quote-stock-b-api/kline
  ?token=YOUR_TOKEN
  &query={"trace":"id","data":{"code":"AAPL.US","kline_type":5,"query_kline_num":300}}

kline_type is the interval in minutes: 1=1m, 5=5m, 15=15m, 30=30m, 60=1h, 1440=1d.

Fallback Aggregation

If AllTick does not provide the requested interval (common with HK instruments on lower API tiers), the system automatically:

  1. Fetches a finer-grained interval (e.g., 1m data for a 5m request).
  2. Validates the cadence of returned data.
  3. Aggregates candles into the target interval using OHLCV bucket logic.

TimescaleDB Candle Storage

The lib/timescale.ts module manages all TimescaleDB interactions:

FunctionSQL OperationPurpose
getNewestFetchedAt() SELECT MAX(fetched_at) Check when candles were last cached (freshness check)
readCandles() SELECT ... WHERE time >= cutoff ORDER BY time ASC LIMIT n Read cached candles from hypertable
writeCandles() INSERT ... SELECT unnest($4::integer[]),... ON CONFLICT DO NOTHING Bulk insert using PostgreSQL unnest() arrays; skips duplicates

Candle Table Schema (TimescaleDB)

CREATE TABLE "Candle" (
  market   TEXT    NOT NULL,    -- 'india' | 'us' | 'hk'
  symbol   TEXT    NOT NULL,    -- instrument code
  interval TEXT    NOT NULL,    -- '1m' | '5m' | ... | '1d'
  time     INTEGER NOT NULL,    -- Unix timestamp (seconds)
  open     NUMERIC NOT NULL,
  high     NUMERIC NOT NULL,
  low      NUMERIC NOT NULL,
  close    NUMERIC NOT NULL,
  volume   NUMERIC NOT NULL,
  fetched_at TIMESTAMPTZ DEFAULT now(),
  PRIMARY KEY (market, symbol, interval, time)
);

Deployment (Vercel)
Deploying MarketSync to production on Vercel

Deployment Prerequisites

  • Vercel CLI installed: npm install -g vercel
  • All databases created and initialized (Prisma Postgres + TimescaleDB)
  • API tokens for Upstox and AllTick obtained

Step-by-Step Deployment

  1. Link to Vercel project
    vercel link (follow prompts to connect to existing project)
  2. Set environment variables via CLI 
    echo "postgres://..." | vercel env add DATABASE_URL production
echo "postgres://..." | vercel env add DIRECT_DATABASE_URL production
echo "your-jwt-secret" | vercel env add JWT_SECRET production
echo "postgres://timescale..." | vercel env add TIMESCALE_URL production
echo "upstox-token" | vercel env add UPSTOX_ACCESS_TOKEN production
echo "alltick-token" | vercel env add NEXT_PUBLIC_ALLTICK_TOKEN production
  3. Deploy to production
    vercel --prod

Build Script

The package.json build command is:

"build": "prisma generate && next build"

This ensures the Prisma client is generated from the schema before the Next.js build runs. The generated client is not committed to git — it is regenerated at build time on every deploy.

Vercel Environment Variables Summary

🔒
Security Warning

NEXT_PUBLIC_ prefixed variables are exposed to the browser. The AllTick token uses this prefix because it is needed in client-side WebSocket connections. Never use NEXT_PUBLIC_ for sensitive secrets like JWT_SECRET or database credentials.

Updating Upstox Token

The Upstox access token expires daily. To refresh it:

  1. Complete the Upstox OAuth flow (click Connect Upstox on the India page).
  2. The new token is received and stored server-side.
  3. Update the UPSTOX_ACCESS_TOKEN in Vercel: vercel env rm UPSTOX_ACCESS_TOKEN production then re-add.

Troubleshooting
Common issues and how to resolve them
SymptomLikely CauseSolution
Blank candlestick chart (intraday) NSE market is closed; intraday returns no candles Chart auto-falls back to 1d interval. This is expected behavior outside 9:15–15:30 IST.
HTTP 502 on chart load Upstox / AllTick API returned an error Chart falls back to 1d interval automatically. Check UPSTOX_ACCESS_TOKEN validity. Upstox tokens expire daily.
India watchlist shows no prices Upstox token missing or expired Click Connect Upstox button and complete OAuth. Update UPSTOX_ACCESS_TOKEN env var.
US/HK watchlist shows no prices AllTick token invalid or quota exceeded Verify NEXT_PUBLIC_ALLTICK_TOKEN in .env. Check AllTick dashboard for quota usage.
Prisma migration error (P3018) A previous migration failed partially Run npx prisma migrate resolve --rolled-back <migration_id> then retry npx prisma migrate deploy.
Module not found: ./generated/prisma/client Prisma generate not run before build Ensure build script is "prisma generate && next build" in package.json.
Candles not caching to TimescaleDB TIMESCALE_URL is empty or invalid Set valid TimescaleDB connection string. Run scripts/init-timescale.sql to create the Candle hypertable.
Order rejected with "Insufficient balance" Cash balance too low for the trade Reduce quantity or check portfolio balance. Starting balance is ₹1,00,000 / $100,000.
SELL order rejected No open position for that symbol Ensure you have previously bought the instrument. Check /portfolio for open positions.
Login redirect loop JWT_SECRET mismatch between env and stored cookies Clear browser cookies. Ensure JWT_SECRET is consistent across deploys.
ConnectionBadge shows Disconnected WebSocket connection dropped The client auto-reconnects. If persistent, refresh the page. Check network / firewall for WSS blocking.
PowerShell: taskkill /PID error Git Bash mangles /PID flag Use: powershell -Command "Stop-Process -Id <PID> -Force"
